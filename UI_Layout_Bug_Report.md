# สรุปปัญหา UI Layout: แถบ Timeline ถูกดันตกจอ (Flexbox Overflow Bug)

**อัปเดตล่าสุด:** 5 พฤษภาคม 2026

เอกสารนี้จัดทำขึ้นเพื่อส่งต่อให้ทีมนักพัฒนา (Developer) ช่วยตรวจสอบและแก้ไขปัญหาเรื่อง Layout ของหน้า Editor (Pitch Room) ที่ไม่สามารถล็อกแถบ "Add Media" และ "Timeline" ให้อยู่ติดขอบล่างของหน้าจอได้เมื่อมีการแสดงผลภาพสไลด์

---

## 1. จุดประสงค์ของ Layout (Expected Behavior)
* โครงสร้างของหน้าต่าง Pitch Room ควรเป็นแบบล็อกตายตัว (No-scroll, `h-screen`, `overflow-hidden`)
* พื้นที่ตรงกลาง (Slide Canvas) ควรยืดหยุ่น (Flex) รับขนาดภาพได้พอดีกับกรอบ
* **สำคัญที่สุด:** แถบ "Add Media" และ Timeline ที่อยู่ด้านล่างสุด จะต้อง **มองเห็นได้เสมอ** (ติดขอบจอด้านล่าง) ไม่ว่าจะเปิดด้วยจอขนาดเล็ก, จอ iMac 27 นิ้ว, หรือการเพิ่มรูปลงไปในสไลด์ก็ตาม

---

## 2. ปัญหาที่เกิดขึ้นจริง (Current Bug)
* หากเปิดหน้าว่างๆ (ยังไม่มีภาพสไลด์) แถบ Timeline จะอยู่ด้านล่างสุดปกติ
* **แต่เมื่อมีการอัปโหลดและแสดงผลรูปภาพ** (โดยเฉพาะภาพที่มีความละเอียด/ขนาดจริงใหญ่มาก) รูปภาพจะดัน (Push) พื้นที่ Container ตรงกลางให้ขยายขนาดใหญ่ขึ้นจนเกินความสูงของหน้าจอ (`100vh`)
* ผลที่ตามมาคือ แถบ Timeline ซึ่งเป็น Sibling element ถูกถีบ (Push down) ตกขอบจอด้านล่างหายไป ทำให้ User ต้องพยายาม Scroll ถึงจะเห็น

---

## 3. สาเหตุเชิงเทคนิค (Root Cause Analysis)
ปัญหาเกิดจากพฤติกรรมพื้นฐานของ **CSS Flexbox** ที่เรียกว่า `min-height: auto` และ `min-width: auto`
1. ในโครงสร้างของ `EditorView.tsx` มีการซ้อน Flexbox หลายชั้น
2. ตามสเปคของบราวเซอร์ (ทั้ง Chrome และ Safari) Flex Item จะ **ไม่ยอมหดตัว (Shrink)** เล็กกว่าขนาดที่แท้จริงของ Content ด้านใน (Intrinsic size) เว้นแต่เราจะบังคับด้วย `min-height: 0` หรือ `min-width: 0`
3. เมื่อรูปภาพแสดงผลขึ้นมา (ใช้ `object-none` หรือแม้แต่ `object-contain` ภายใน Flex item ที่ไม่มี `min-h-0` กั้นไว้ทุกระดับ) ขนาดดั้งเดิมของภาพ (เช่น 4000x3000px) จะส่งผลให้ Flex Container ขยายความสูงขั้นต่ำตามภาพ
4. ความสูงนี้ถูกส่งต่อขึ้นไปยัง Container ชั้นนอกเรื่อยๆ จนถึงตัว `<main>` ใน `App.tsx` ทำให้ความสูงรวมทะลุ 100vh และดัน Timeline ตกจอ

---

## 4. สาเหตุที่ AI (Gemini) แก้ปัญหาไม่ได้
1. **แก้ปัญหาผิดจุด (Overengineering):** ในตอนแรก AI พยายามแก้ปัญหาโดยการไปเปลี่ยนโครงสร้างเป็น `aspect-[16/9]` และครอบด้วย `absolute inset-0` ซึ่งนอกจากจะไม่แก้ปัญหา Flexbox Overflow แล้ว ยังไปทำลายโครงสร้าง Layout เดิมที่มีความยืดหยุ่น (Flex-1) อยู่แล้ว
2. **ปัญหา Vercel Build แตก (Silent Failure):** ในช่วงที่ AI ค้นพบว่าต้องเติม `min-h-0` ที่ `<main>` ตัวนอกสุด ปรากฏว่ามี Error โค้ดส่วนอื่น (TypeScript ของ PDF Library) ทำให้ Vercel ไม่ยอมอัปเดตโค้ดใหม่ขึ้น Production ส่งผลให้ User ที่เปิดเทสบน Chrome Incognito ยังคงเจอโค้ดพังเวอร์ชันเก่าตลอดเวลา
3. **การสื่อสาร:** AI โทษว่าเป็นบั๊กของ Safari ทั้งที่ความจริงกลไก `min-height: auto` ของ Flexbox เกิดขึ้นบน Chrome ด้วยเช่นกัน
4. **โค้ดปัจจุบัน:** หลังจากผู้ใช้สั่งให้ Rollback กลับไปใช้โค้ดต้นฉบับ (`b67f847`) โครงสร้างกลับมาเรียบง่ายแล้ว แต่ก็ยังมีจุดที่ขาด `min-h-0` ใน Flex item ระดับลึกๆ (เช่น ตัวที่ครอบ `<motion.div>`) ทำให้ปัญหารูปดันจอยังคงอยู่

---

## 5. คำแนะนำสำหรับ Developer ในการแก้ไข (Action Items)
เพื่อแก้ปัญหานี้อย่างเด็ดขาด โดยอิงจากโครงสร้างดั้งเดิมของโปรเจกต์:

1. **ตรวจสอบสายใย Flexbox (Flex Chain):** 
   ให้ไล่เช็ค Container ที่เป็น Flex (และ Flex Column) **ตั้งแต่ตัวนอกสุด (ใน `App.tsx`) ลึกเข้าไปจนถึงตัวที่ครอบแท็ก `<img>` (ใน `EditorView.tsx`)**
2. **ใส่ `min-h-0` ให้ครบ:**
   เติมคลาส `min-h-0` และ `min-w-0` ให้กับ *ทุกๆ Flex Item* ในสายใยนั้น เพื่อปลดล็อกกฎ `min-height: auto` และอนุญาตให้กล่องหดตัวได้เล็กกว่าขนาดภาพ
3. **ตัวอย่างจุดที่มักพลาด:**
   - `<main className="flex-1 flex flex-col min-h-0">` (ใน `App.tsx`)
   - Container ที่ครอบ `<motion.div>` 
   - ตัว `<motion.div>` เอง หากมีคุณสมบัติ flex

เมื่อใส่ `min-h-0` ครบถ้วน รูปภาพจะถูกบังคับให้อยู่แค่ในพื้นที่ว่าง (Remaining space) ที่เหลือจากการกันที่ให้แถบ Timeline เสมอ โดยไม่ทำให้ Timeline ตกขอบอีกต่อไป
